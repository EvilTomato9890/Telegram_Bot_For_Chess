# Telegram Bot for Chess

Telegram-бот для проведения шахматных турниров по швейцарской системе.

## Требования

- **Python**: `3.11+` (см. `requires-python` в проекте).
- **База данных**:
  - по умолчанию `SQLite` через `aiosqlite`;
  - также можно использовать любую БД, поддерживаемую SQLAlchemy async driver, если корректно задан `DATABASE_URL`.
- **Основные зависимости**:
  - `aiogram` — Telegram Bot API;
  - `sqlalchemy` — ORM и запросы;
  - `alembic` — миграции схемы;
  - `aiosqlite` — async-драйвер для SQLite;
  - `pydantic-settings` — конфигурация из окружения;
  - `loguru` — логирование.
- **Dev-зависимости**: `pytest`, `ruff`, `mypy`.

## Пример `.env`

Скопируйте `.env.example` в `.env` и заполните значения:

```env
TOKEN=123456:telegram-token
ADMIN_IDS=123456789,987654321
TIMEZONE=UTC
DATABASE_URL=sqlite+aiosqlite:///./tournaments.db

# Флаги поведения (допущения для текущего ТЗ, см. раздел "Допущения")
AUTO_ADVANCE_TO_NEXT_ROUND=false
MODERATE_RESULTS=true

LOG_LEVEL=INFO
```

### Назначение переменных

- `TOKEN` — токен Telegram-бота.
- `ADMIN_IDS` — CSV-список Telegram user id организаторов.
- `TIMEZONE` — IANA timezone (например, `Europe/Moscow`).
- `DATABASE_URL` — строка подключения SQLAlchemy.
- `AUTO_ADVANCE_TO_NEXT_ROUND` — автоматический переход к новому туру после завершения предыдущего.
- `MODERATE_RESULTS` — требование подтверждения/модерации результатов.
- `LOG_LEVEL` — уровень логирования.

## Установка и запуск

### 1) Установка зависимостей

```bash
python -m venv .venv
source .venv/bin/activate
pip install -U pip
pip install -e .[dev]
```

### 2) Миграции Alembic

```bash
alembic upgrade head
```

> Если нужно откатиться на одну миграцию:
>
> ```bash
> alembic downgrade -1
> ```

### 3) Запуск бота

```bash
python -m bot.main
```

## Минимальные end-to-end сценарии

Ниже — минимальный smoke-путь для ручной проверки бизнес-потока.

1. **Регистрация игроков и столов**
   - Организатор создает турнир.
   - Регистрируются минимум 4 игрока.
   - Создаются столы (по числу партий в туре).
   - Проверка: игрок по `/my_next` до старта тура не должен получить активную партию.

2. **Старт турнира**
   - Турнир переводится в статус `ACTIVE`.
   - Установлены правила (`/rules`) и расписание (`/schedule`).
   - Проверка: игроки видят правила и расписание.

3. **Генерация туров**
   - Организатор запускает генерацию первого тура (`/pair_round`).
   - Проверка: каждому игроку назначена партия/`BYE`, `/my_next` возвращает актуальную игру.

4. **Репорт / аппрув результатов**
   - Игрок A отправляет `/report <round> <result>`.
   - Игрок B отправляет совпадающий результат — партия фиксируется как `FINISHED`.
   - Если результаты не совпадают, партия остается на модерацию организатора.

5. **Завершение турнира и итоговая таблица**
   - После последнего тура турнир завершается.
   - Проверка: `/standings` показывает итоговый рейтинг в корректном порядке.

## Допущения и принятые решения

Чтобы снять неоднозначности ТЗ, зафиксированы следующие правила:

1. **Модерация результатов включена по умолчанию**:
   - первый репорт помечается как pending;
   - совпадающий репорт соперника автоматически подтверждает результат;
   - конфликтующие репорты эскалируются организатору.

2. **Автопереход к следующему туру по умолчанию выключен**:
   - следующий тур стартует явной командой организатора.

3. **BYE-партия**:
   - репорт от игрока не требуется.

4. **Ограничение статусов**:
   - нельзя генерировать новый тур, пока предыдущий не завершен.

5. **Формат результата репорта**:
   - поддерживаются `1-0`, `0-1`, `0.5-0.5` (и синоним `1/2-1/2`).

## Tie-break правила

Сортировка таблицы фиксирована и выполняется строго в порядке:

1. **Points**
2. **Buchholz** (сумма очков соперников)
3. **Sonneborn-Berger** (сумма очков соперников, умноженных на набранный против них результат)
4. **Median Buchholz**
5. **Player ID (ASC)** — технический детерминирующий tie-break при полном равенстве.
